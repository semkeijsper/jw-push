# CLAUDE.md

This file provides guidance for AI assistants working in this codebase.

## Project Overview

`jw-push` is a Node.js bot that polls JW.ORG APIs and RSS feeds, then pushes new content (videos, articles, news alerts) to WhatsApp channels using the `whatsapp-web.js` library. It supports multiple channels simultaneously, each with its own language/locale configuration.

## Tech Stack

- **Runtime**: Node.js >= 24
- **Language**: TypeScript 6 (strict mode, ES modules)
- **Package manager**: pnpm 10
- **WhatsApp**: `whatsapp-web.js` (Puppeteer-based)
- **Module system**: ESM (`"type": "module"`)
- **Build output**: `dist/` (compiled from `src/`)

## Commands

```bash
pnpm dev              # Run with tsx (no build needed, for development)
pnpm build            # Compile TypeScript to dist/
pnpm start            # Run compiled dist/index.js
pnpm lint             # ESLint on src/
```

**CLI flags** passed to `node`/`tsx`:
- `--baseline` — Mark all current JW.ORG content as already sent (no messages sent). Use on first run to avoid flooding channels.
- `--force` — Clear state and resend all content. Only affects `development` channels.
- `--linux` — Use `/usr/bin/chromium` instead of bundled Chromium (for Linux servers).

## Repository Structure

```
src/
  index.ts      Entry point: WhatsApp client setup, reads args, spawns JWBot per channel
  bot.ts        JWBot class: polling loops (60s), deduplication, send/edit/delete logic
  api.ts        JW.ORG API and RSS feed fetching with retry logic
  format.ts     Message formatting (video, article, alert) with HTML-to-WhatsApp conversion
  state.ts      BotState class: file-backed GUID tracking (run/{channelId}.json)
  config.ts     Loads config.json and exports typed channel list
  types.ts      All TypeScript type definitions
  i18n.ts       Localization strings (currently: en, nl)
  logger.ts     Channel-prefixed console logger utility
dist/           Compiled JS (git-ignored, generated by `pnpm build`)
run/            Runtime state files per channel (git-ignored)
.wwebjs_auth/   WhatsApp session cache (git-ignored, persists login across restarts)
config.json     Active configuration (git-ignored, see config.example.json)
```

## Configuration

Copy `config.example.json` to `config.json` and fill in channel IDs:

```json
{
  "channels": [
    {
      "id": "1234567890@newsletter",
      "type": "production",
      "name": "English",
      "langcode": "E",
      "locale": "en",
      "articleFeedUrl": "https://www.jw.org/en/whats-new/rss/WhatsNewWebArticles/feed.xml"
    }
  ]
}
```

- `id`: WhatsApp channel ID in the format `{id}@newsletter`
- `type`: `"production"` or `"development"` (only `development` is affected by `--force`)
- `langcode`: JW.ORG language code (e.g., `"E"` for English, `"O"` for Dutch)
- `locale`: Locale string for formatting and i18n (e.g., `"en"`, `"nl"`)
- `articleFeedUrl`: Full RSS feed URL for the language-specific "What's New?" feed

## Code Conventions

These are enforced by ESLint (`pnpm lint`):

- **Semicolons**: always required
- **Quotes**: double quotes only (`"string"`, not `'string'`)
- **Braces**: always explicit — never omit `{}` on `if`/`for`/`while` blocks
- **Brace style**: Stroustrup — `else`/`catch` go on a new line after `}`
- **Indentation**: 4 spaces; `case` labels indented 4 spaces inside `switch` (8 total)
- **Type definitions**: use `type` keyword, not `interface`
- **TypeScript**: strict mode; all `@typescript-eslint/recommendedTypeChecked` and `stylisticTypeChecked` rules apply
- **Imports**: use `.js` extension in import paths (ESM with NodeNext resolution)
- **`void`**: prefix floating promises with `void` (e.g., `void bot.start(...)`)

Example of correct brace style (Stroustrup):

```typescript
if (condition) {
    doSomething();
}
else {
    doOther();
}
```

## Architecture Patterns

### Polling

Each `JWBot` instance runs three independent polling loops with staggered starts to avoid bursting all requests at once:

| Content | Start delay | Interval |
|---------|-------------|----------|
| Videos  | 0s          | 60s      |
| Articles| 15s         | 60s      |
| Alerts  | 30s         | 60s      |

### Deduplication

`BotState` tracks sent GUIDs in a JSON file at `run/{channelId}.json`. Content is only sent if its GUID has not been seen before. Each list is capped at 500 entries to prevent unbounded growth.

Articles are deduplicated by **link URL**, not GUID — RSS can republish the same article with a different GUID, and `BotState.itemKey()` normalises this by using the link as the canonical key.

### Universal Alerts

JW.ORG may publish a "universal" (`languageCode === "univ"`) alert before the localized version is available. The bot:
1. Sends the universal alert immediately and stores the `Message` object in memory.
2. When the localized version arrives (same GUID, different `languageCode`), edits the original message in place.
3. Falls back to delete-and-resend if the edit window has expired or the message is not in memory.

Editing a message in-place produces no new channel notification, so the 2-second inter-message delay is skipped after an edit (see "Message Rate Limiting" below).

### API Retry

All JW.ORG API calls retry up to 5 times with a 500ms delay between attempts. 503 responses are treated as empty results (safe fallback).

### Message Rate Limiting

A 2-second delay is inserted between consecutive sends to avoid WhatsApp rate-limiting. The delay is skipped when a universal alert is edited in-place (edits don't produce channel notifications, so no throttle is needed).

### Concurrent Poll Protection

Each polling loop is guarded by a boolean mutex (`checkingVideos`, `checkingArticles`, `checkingAlerts`). If a previous check is still running when the next interval fires, the new tick is skipped. This prevents overlapping API calls and duplicate sends under slow network conditions.

### Video Thumbnails

Videos are sent with an image thumbnail when available. The thumbnail URL is selected in priority order: `lsr.xl` → `lss.lg` → `wss.lg` → `sqr.lg`. If fetching the image fails, the bot falls back to sending text only.

### Message Formatting

`src/format.ts` converts JW.ORG API data into WhatsApp-formatted strings:

- `htmlToWhatsApp(html)` — converts `<strong>` → `*bold*`, `<em>` → `_italic_`, `<br/>` → newlines (used for alert bodies)
- `stripHtml(html)` — removes all HTML tags without converting them (used for alert titles)
- `boldBeforePipe(text)` — if the text contains `|`, wraps the part before the pipe in `*bold*` (used for article and alert titles where the feed uses `|` as a title separator)
- `getVideoUrl(video, locale)` — constructs the canonical `jw.org/finder` URL from `languageAgnosticNaturalKey` and locale

## State Files

State is stored in `run/{channelId}.json` (git-ignored). This file is created automatically and contains:

```json
{
  "pushedVideos": ["guid1", "guid2"],
  "pushedAlerts": ["guid1"],
  "pushedArticles": ["guid1"],
  "univAlerts": { "guid1": "serialized_message_id" }
}
```

Do not commit `run/` or `config.json`.

## Adding a New Language

1. Add a new entry to the `strings` map in `src/i18n.ts` with a locale key.
2. Add a matching channel entry in `config.json` with that locale.
3. The bot will automatically use the correct strings; English is the fallback.

## No Test Suite

There is no test framework configured. The `test` script in `package.json` is a placeholder. Validate changes by running `pnpm lint` and `pnpm build`, then testing manually with `pnpm dev --baseline`.
